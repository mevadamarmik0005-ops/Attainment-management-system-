# CO/PO Attainment System — KDP, Patan

A full-stack final-year project: a login-gated web app that calculates Course Outcome (CO)
and Program Outcome / PSO (PO/PSO) attainment (NBA-style), with Excel upload, manual entry,
a results dashboard, and a **real backend + database** (not a demo/mock).

- **Backend:** Node.js + Express
- **Auth:** username/password, bcrypt-hashed, JWT sessions
- **Frontend:** React, bundled with esbuild, served by the same Express server
- **Database: two interchangeable drivers, switch with one line in `.env`:**
  - `DB_DRIVER=json` (default) — a dependency-free local file (`data/attainment.json`).
    Zero setup, but only reachable from the computer running the server.
  - `DB_DRIVER=mongo` — MongoDB (e.g. a free MongoDB Atlas cluster). A real network
    database: run the server on your laptop, a classmate's laptop, or a free host, and
    as long as everyone uses the same `MONGODB_URI` they all read/write the same live data.

> **Why not SQLite?** An earlier version used `better-sqlite3`, but that's a *native*
> module — it needs a C++ compiler (Python + build tools) to install, which fails on a
> lot of Windows machines that don't have those set up. The `json` driver here
> (`src/db/json.js`) is hand-written with zero native dependencies, so `npm install`
> never needs to compile anything, while still behaving like a real database (separate
> `users`/`courses` tables, per-user isolation, atomic writes). The `mongo` driver
> (`src/db/mongo.js`) is there for when you specifically need multiple computers to
> share one live database. Both implement the exact same functions, so the rest of the
> app (routes, auth, frontend) doesn't know or care which one is active.

## 2. If `npm install` or `npm start` fails

- **"node is not recognized" / command not found:** install Node.js 18+ from
  https://nodejs.org (the LTS version), then reopen your terminal.
- **Port already in use:** another process is using port 4000. Either stop it, or run
  the server on a different port: `PORT=5000 npm start` (Windows PowerShell:
  `$env:PORT=5000; npm start`).
- **`npm run build` fails with a esbuild/module error:** delete `node_modules` and
  `package-lock.json`, then run `npm install` again.
- **Login page loads but "Sign In" does nothing / network error:** make sure the
  server is actually running (`npm start`) and you're visiting
  `http://localhost:4000`, not opening `public/index.html` directly as a file — the
  frontend needs the API, which is only reachable through the running server.
- Still stuck? Copy the exact error text from your terminal — that's what tells us
  what actually broke.

## 5. How the attainment calculation works

- **Direct CO Attainment** = Internal Weight% × average(Internal-1, Internal-2, Assignment levels)
  + End-Sem Weight% × End-Sem level
- **Overall CO Attainment** = Direct Weight% × Direct Attainment + Indirect Weight% × Indirect Attainment
  (Indirect = average student self-rating from the course-exit survey)
- **PO/PSO Attainment** = weighted average of the six COs' final attainment, weighted by the
  CO→PO/PSO mapping strength (1–3) entered in the Mapping tab
- All weights and thresholds are editable in the **Setup** tab.

Each internal exam / assignment has 6 questions, one per CO (Q1→CO1 … Q6→CO6).
The End-Sem exam has 12 questions — Q*i* and Q*(i+6)* both test CO*i*.

## 6. Excel import / export

- **Upload Excel** parses a `.xlsx` file with sheets named `Course_Info`, `CO_PO_PSO_Mapping`,
  `Internal_Exam1`, `Internal_Exam2`, `Assignment`, `End_Sem_Exam`, `Indirect_CO_Attainment`
  (this matches the companion downloadable Excel workbook from the same project).
- **Export Excel** writes the current on-screen data back out to a workbook in the same layout,
  so you can round-trip between the website and Excel.
- Sheets that aren't found in an uploaded file are simply skipped (existing data for that
  section is kept), and you'll see a banner listing what was and wasn't imported.

## 7. Database & accounts — what's real here

- Every registered user has their **own private** saved courses — nobody else can see them.
- Passwords are hashed with bcrypt before they ever touch the database; the server never
  stores or logs plain-text passwords.
- Multiple people can register their own accounts and each maintain their own set of saved
  courses — this is a genuine multi-user system, not a single shared demo account.
- This app's `/api/health` endpoint reports which database driver is active — useful for
  confirming your `.env` setting took effect (`curl http://localhost:4000/api/health`).

### Option A — local file (`DB_DRIVER=json`, the default)
Data is persisted in `data/attainment.json`. You can open it in any text editor to see
exactly what's stored, or back it up by simply copying the file. This only works from the
one computer running the server — great for development and grading on your own machine.

### Option B — MongoDB (`DB_DRIVER=mongo`), reachable from any computer
Use this when you need the database itself to live on the network, so it doesn't matter
which computer runs `npm start` — everyone using the same connection string sees the same
live data. A free MongoDB Atlas cluster is the easiest way to get this:

1. Go to https://www.mongodb.com/cloud/atlas/register and create a free account.
2. Create a free **M0** cluster (takes a couple of minutes to provision).
3. **Database Access** (left sidebar) → Add New Database User → pick a username/password
   (save these — you'll need them in the connection string).
4. **Network Access** (left sidebar) → Add IP Address → "Allow Access from Anywhere"
   (`0.0.0.0/0`) — fine for a class project; tighten this for anything more sensitive.
5. Click **Connect** on your cluster → **Drivers** → copy the connection string, it looks like:
   `mongodb+srv://<username>:<password>@cluster0.xxxxx.mongodb.net/`
6. In your `.env`:
   ```
   DB_DRIVER=mongo
   MONGODB_URI=mongodb+srv://<username>:<password>@cluster0.xxxxx.mongodb.net/
   MONGODB_DB=attainment
   ```
7. Run `npm run seed` then `npm start` as usual — it'll create the same Admin9 account,
   just inside your Atlas cluster instead of a local file.

Now anyone who clones this repo, adds the **same** `MONGODB_URI` to their own `.env`, and
runs `npm start` on their own computer is working against the exact same shared database —
useful for a viva/demo where more than one person needs to see the same live data, or for
running the server on a free host while still being able to hit the same data from your
own laptop for debugging.

## 8. Deploying it for real (so it has its own public URL)

Right now this runs locally. To make it a live website:

1. **Pick a host** that runs Node.js processes, e.g. Render, Railway, or a small VPS.
   - Using `DB_DRIVER=json`: pick a host with a **persistent disk** (Render/Railway/VPS all
     support this) — pure "serverless" hosts wipe local files between requests, which would
     lose your data.
   - Using `DB_DRIVER=mongo`: the database lives on MongoDB Atlas, not on the host's disk,
     so serverless hosts work fine too — this is the easier option to deploy.
2. Push this project to a GitHub repo (see section 10 below).
3. On the host, set the **build command** to `npm install && npm run build` and the
   **start command** to `npm start`.
4. Set environment variables on the host: `JWT_SECRET` (a long random string), and either
   `DATA_DIR` (json driver) or `MONGODB_URI` + `MONGODB_DB` (mongo driver).
5. Run `npm run seed` once (via the host's shell/console) if you want the Admin9 demo login
   available in production too — or just register a real account through the UI.

If your college requires MySQL/PostgreSQL specifically instead of MongoDB, add a third file
in `src/db/` implementing the same functions as `json.js`/`mongo.js`, and point `DB_DRIVER`
at it in `src/db/index.js` — the rest of the app (auth, API shape, frontend) stays the same.

## 9. Security notes for submission / viva

- Passwords: bcrypt, salted automatically, 10 rounds.
- Sessions: signed JWTs (30-day expiry), verified on every protected request.
- Each user can only read/write their **own** courses — enforced server-side on every query
  (`WHERE user_id = ?`), not just hidden in the UI.
- `.env` (with your real `JWT_SECRET`) is git-ignored — never commit it.
- For a public-facing deployment, also add HTTPS (most hosts do this for you) and consider
  rate-limiting the `/api/auth/*` routes against brute-force login attempts.

## 10. Committing this to GitHub

This project already has a `.gitignore` set up correctly (it excludes `node_modules/`,
`data/`, `public/bundle.js`, and `.env` — you never want to commit any of those).

```bash
git init                                   # skip if this is already a git repo
git add .
git commit -m "CO/PO Attainment System - initial commit"
git branch -M main
git remote add origin https://github.com/<your-username>/<your-repo-name>.git
git push -u origin main
```

A few things worth doing before you push:
- Double check `git status` doesn't list `.env`, `node_modules/`, or `data/` — if it does,
  something's off with `.gitignore` (they should never appear).
- `public/bundle.js` is also excluded on purpose (it's a build artifact) — anyone who clones
  the repo just needs to run `npm install && npm run build` once to regenerate it.
- If you used the MongoDB driver, **never** commit your real `MONGODB_URI` — it contains
  your database password. Keep it only in your local `.env` (or in your host's environment
  variable settings when you deploy).
